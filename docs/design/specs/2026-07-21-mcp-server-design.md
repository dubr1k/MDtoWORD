# MCP-сервер для MDtoWORD

**Дата:** 2026-07-21
**Статус:** согласован, готов к планированию

## Задача

Дать агентам (Claude Code, Codex, любой MCP-клиент) программный доступ к конвертации
Markdown → Word и Word → Markdown, которая сегодня доступна только через GUI.

## Решения, принятые при обсуждении

| Вопрос | Решение |
|---|---|
| Контракт I/O | Только пути к файлам. Ни base64, ни inline-контента ни в одну сторону. |
| Объём API | Три инструмента: две конвертации + preview/dry-run. |
| Запуск | stdio, из этого же репозитория. |
| Политика записи | Без ограничений: пишем по указанному пути, перезаписываем молча. |

## Ограничение, принятое осознанно

Перезапись не защищена. При вызове `markdown_to_word` по папке, где уже лежат `.docx`
с совпадающими именами, они будут заменены без предупреждения и без возможности отката.
Ответственность за подтверждение — на MCP-клиенте.

## Архитектура

Сегодня `MarkdownToWordConverter` и `WordToMarkdownConverter` объявлены внутри
`mdtoword/app.py`, который в строках 7–14 импортирует PyQt6. Любой headless-потребитель
конвертеров тянет за собой Qt. Это блокирует MCP-сервер и уже мешает тестам:
`tests/test_markdown_converter.py` требует PyQt6 для проверки чистой логики рендеринга.

Ядро выносится в Qt-free модуль:

```
mdtoword/
  converters.py    НОВОЕ  ConversionError, MarkdownToWordConverter, WordToMarkdownConverter
  mcp_server.py    НОВОЕ  три инструмента поверх converters + workflow
  app.py           GUI; импортирует конвертеры, локализует ошибки
  gfm_renderer.py  без изменений
  latex_omml.py    без изменений
  workflow.py      без изменений
  theme.py         без изменений
```

Зависимости строго односторонние: `mcp_server.py → converters.py → gfm_renderer.py → latex_omml.py`.
PyQt6 не встречается ни в одном звене этой цепочки — это и делает сервер запускаемым
без дисплея.

### Контракт ошибок

Текущие конвертеры возвращают `(bool, str)`, где строка — это `"Ошибка при конвертации: ..."`,
а варнинги приклеены к тому же сообщению через `\n\n`. То есть локализация зашита в ядро,
а варнинги не отделимы от текста программно. Для батча в MCP это непригодно.

Новый контракт ядра:

- успех → возврат `list[str]` варнингов (возможно пустого);
- отказ → исключение `ConversionError` с сообщением без локали.

Локализация переезжает в `app.py`, где уже есть словарь `_text()` с RU/EN. GUI оборачивает
вызовы в `try/except ConversionError` и форматирует сообщение сам.

### Транспорт

Официальный Python SDK `mcp`, декораторный сервер, транспорт stdio.
Structured output — через Pydantic-модели.

**stdout — это сам протокол.** Любой `print` в цепочке импорта или выполнения рвёт сессию.
Логирование только в stderr. Защищено тестом.

### Установка

`pyproject.toml` в репозитории нет, console script без него не сделать, и заводить его
ради одного скрипта избыточно. Конфигурация клиента:

```json
{
  "command": "<repo>/.venv/bin/python",
  "args": ["-m", "mdtoword.mcp_server"],
  "cwd": "<repo>"
}
```

SDK `mcp` выносится в отдельный `requirements-mcp.txt`, а не в `requirements.txt`.
Причина: `requirements.txt` — источник истины для PyInstaller-сборки GUI и для
`test_packaging.py`, который сверяет его с `environment.yml`. Добавление туда серверного
SDK раздуло бы дистрибутив приложения и потребовало правки обоих файлов ради зависимости,
которая GUI не нужна.

## Инструменты

Общая форма отчёта у всех трёх — агенту не нужно учить три разные схемы.

| Инструмент | Параметры | Возврат |
|---|---|---|
| `markdown_to_word` | `inputs: list[str]`, `output_dir: str \| None`, `font_name: str = "Times New Roman"`, `font_size: float = 12`, `footnotes_heading: str = "Footnotes"` | `ConversionReport` |
| `word_to_markdown` | `inputs: list[str]`, `output_dir: str \| None` | `ConversionReport` |
| `preview_markdown` | `inputs: list[str]`, `font_name: str = "Times New Roman"`, `font_size: float = 12` | `PreviewReport` |

```
ConversionReport
  sources_found: int
  converted: list[{ source: str, output: str, warnings: list[str] }]
  failed:    list[{ source: str, error: str }]

PreviewReport
  sources_found: int
  previews: list[{ source: str, warnings: list[str] }]
  failed:   list[{ source: str, error: str }]
```

`preview_markdown` прогоняет полный рендер в память и не пишет ничего на диск. Нужен,
чтобы агент увидел предупреждения о непереводимых конструкциях (LaTeX, недоступные
изображения) и починил исходник до записи файла.

`inputs` принимает файлы и папки вперемешку: рекурсивный обход и фильтрация по
расширениям уже реализованы в `workflow.discover_sources`. Имена выходных файлов
выделяет `workflow.resolve_output_paths`, он же разводит коллизии внутри батча.

### Поведение на краях

- **`inputs` пустой** → `ConversionError` с явным сообщением, а не пустой успешный отчёт.
- **`discover_sources` ничего не нашёл** → `sources_found: 0`, списки пустые. Поле
  существует именно затем, чтобы агент отличил «путь указан неверно» от «всё готово».
  Молчаливый пустой успех — худший из возможных ответов агенту.
- **Падение одного файла** не прерывает батч: файл уходит в `failed`, остальные
  конвертируются.
- **`output_dir` не существует** → создаётся с `parents=True`.
- **Существующий выходной файл** → перезаписывается молча (см. «Ограничение, принятое
  осознанно»).

### Описания инструментов

Описания — единственная документация, которую видит агент, поэтому в них прямым текстом:

- `markdown_to_word` поддерживает GFM, таблицы, сноски, изображения и LaTeX → OMML
  (нативные формулы Word);
- `word_to_markdown` **лоссовый**: извлекает заголовки по стилям `Heading N`,
  bold/italic по run'ам и таблицы; всё остальное — списки, изображения, формулы,
  сноски, стили — схлопывается в плоский текст.

Без явного предупреждения о потерях агенты будут гонять round-trip `md → docx → md`
и терять содержимое.

## Тестирование

- `tests/test_converters.py` — ядро без Qt. Существующий `tests/test_markdown_converter.py`
  переезжает сюда с импортом из `mdtoword.converters`; проверки не меняются.
- `tests/test_mcp_server.py` — in-memory клиент из SDK: схемы трёх инструментов,
  батч по папке, частичный отказ (один битый файл среди валидных), `sources_found: 0`
  на несуществующем пути, пустой `inputs`.
- Тест «импорт `mdtoword.mcp_server` ничего не пишет в stdout» — защита stdio-протокола.
- Регресс: остальные шесть наборов проходят без изменений.

## Что осознанно не делается

- Sandbox по разрешённым корням и флаг `overwrite` — отклонено при обсуждении.
- HTTP-транспорт — бессмысленен при контракте «только пути»: файловая система всё равно
  должна быть общей.
- Публикация отдельного пакета на PyPI — не нужна, пока потребитель один.
- Улучшение качества `word_to_markdown` — отдельная задача, не входит в этот спек.
  Здесь его лоссовость только документируется.
