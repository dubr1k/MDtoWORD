# MCP-сервер для MDtoWORD

**Дата:** 2026-07-21
**Статус:** согласован, готов к планированию

## Задача

Дать агентам (Claude Code, Codex, любой MCP-клиент) программный доступ к конвертации
Markdown → Word и Word → Markdown, которая сегодня доступна только через GUI.

## Решения, принятые при обсуждении

| Вопрос | Решение |
|---|---|
| Контракт I/O | Только пути к файлам. Ни base64, ни inline-контента ни в одну сторону. |
| Объём API | Три инструмента: две конвертации + preview/dry-run. |
| Запуск | stdio, из этого же репозитория. |
| Политика записи | Без ограничений: пишем по указанному пути, перезаписываем молча. |
| Сетевой доступ при чтении | По умолчанию выключен для MCP-сервера (`fetch_remote_images=false`); для GUI остаётся включённым — см. «Сетевой доступ: удалённые изображения». |

## Ограничение, принятое осознанно

Перезапись не защищена. При вызове `markdown_to_word` по папке, где уже лежат `.docx`
с совпадающими именами, они будут заменены без предупреждения и без возможности отката.
Ответственность за подтверждение — на MCP-клиенте.

## Сетевой доступ: удалённые изображения

Этот спек изначально не говорил ничего про сетевой egress: строка «без
ограничений» в таблице выше была про политику записи на диск, а не про
исходящие запросы при чтении. Отдельно фиксируем решение, принятое уже после
первой версии спека.

`gfm_renderer._append_image` умеет получать изображение по `http(s)`-ссылке
через `urlopen`. Для GUI это уместно: файл открыл человек, и он же отвечает
за то, каким ссылкам в нём доверять. Для MCP-сервера предположение не
работает — агента можно направить на чужой Markdown (свежесклонированный
репозиторий, README, найденный в интернете), и тогда сервер становится
исполнителем произвольных сетевых запросов от имени того, кто написал
документ, а не того, кто им управляет: он стучится по указанному в файле
адресу и отчитывается в `warnings`, был ли запрос отклонён, истёк ли таймаут
или адрес оказался достижим. Это рабочий слепой SSRF-зонд по localhost,
локальной сети и облачным metadata-эндпоинтам, использующий сетевую позицию
сервера, а не автора Markdown.

Поэтому у `markdown_to_word` и `preview_markdown` — единственных инструментов,
которые рендерят Markdown, — добавлен параметр `fetch_remote_images: bool =
False`. По умолчанию изображение по `http(s)`-ссылке не загружается и
превращается в альтернативный текст с предупреждением; загрузка включается
только явной передачей `fetch_remote_images=true`, то есть агент должен
подтвердить, что документ — из источника, которому он доверяет. `GfmDocxRenderer`
получает тот же переключатель как `allow_remote_images`, по умолчанию `True`,
так что GUI (`app.py` его не передаёт) продолжает загружать удалённые
изображения без изменений. `word_to_markdown` рендер не делает и этого
параметра не получает.

Опция не ограничивает, куда в итоге уйдёт запрос: `urlopen` безусловно
следует 3xx-редиректам без ограничения на хост или диапазон IP и без защиты
от DNS rebinding, а обработчик редиректов в `urllib` пропускает переход с
`https` на `ftp`. Поэтому включивший `fetch_remote_images=true` для
документа из доверенного источника доверяет не только автору документа, но
и любому редиректору на пути: конечный адрес — например, `127.0.0.1` или
`169.254.169.254` — может подставить третья сторона, и совет «доверяй
источнику» на цель редиректа не распространяется. Обрезать редиректы не
планируется: это сломало бы обычные картинки с CDN.

## Локальные пути: какие изображения читает markdown_to_word

Смежная дыра, симметричная сетевому egress выше, но по чтению локальной
файловой системы, а не по исходящим запросам. Файловая ветка
`_append_image` резолвит `src` из Markdown как обычный путь:
`![x](/Users/victim/private/scan.png)` — абсолютный путь, который попадёт
прямо в выходной `.docx`; `![x](../../../etc/hosts)` — выход за пределы
папки с исходником. К тому же предупреждения различают `Image not found:
X` и `Image could not be rendered: X (...)`, а это оракул существования:
гоняя один и тот же вызов по разным путям, агент узнаёт, какие файлы есть
на диске, даже не читая их содержимое.

Для GUI это не проблема — файл открыл человек, значит человек же и
определил границы. Для MCP-сервера рассуждение не работает по той же
причине, что и с сетевым egress: путь к изображению приходит из тела
документа, который агенту могли подсунуть, а не выбрать самому.

Решение — асимметричный дефолт по образцу `fetch_remote_images`, но для
чтения, а не для сети. `GfmDocxRenderer` получает `image_roots:
Sequence[Path] | None = None`: `None` — без ограничений, ровно текущее
поведение, `app.py` этот параметр не передаёт и разницы не видит.
`MarkdownToWordConverter` пробрасывает его тем же способом, что и
`allow_remote_images`. У `markdown_to_word` и `preview_markdown` корни
строятся из `inputs` — по одному на каждый входной путь: для директории —
сама директория, для файла — его родитель, без свёртки в общий предок.
Общий предок у путей из разных, не пересекающихся веток может схлопнуться
в `/`, и ограничение молча исчезнет — худший исход для контроля
безопасности. Явный параметр `image_root` — предохранительный клапан для
раскладок, где изображения лежат вне `inputs`.

Проверка на принадлежность корню выполняется на **резолвленных** путях
(`Path.resolve()` в нестрогом режиме — работает и для путей, которых не
существует). Сравнения префиксов строк недостаточно: `../../../` уходит из
корня прямым текстом, а символическая ссылка *внутри* разрешённого корня,
указывающая наружу, пройдёт префиксную проверку, но не резолвленную.

Порядок проверки принципиален: она выполняется до `is_file()`, а не после.
Сам факт обращения к файловой системе за ответом «существует или нет» и
есть тот оракул, который нужно закрыть, — поэтому спрашивать диск об этом
для уже отвергнутого пути нельзя. Сообщение при отказе — `Image outside
the allowed root: {target} (pass image_root=... to widen it)` — одинаково
для существующего и несуществующего файла: оно не несёт никакой информации
о результате `is_file()`, потому что до него дело не доходит.

Раздел «Что осознанно не делается» ниже фиксирует отказ от «sandbox по
разрешённым корням» — но то решение было о доступе к файловой системе в
целом заодно с флагом `overwrite`, и принято на этапе, когда ни политика
записи, ни оракул существования при чтении по отдельности ещё не
разбирались. Нынешнее ограничение — не тот отклонённый общий sandbox: оно
узкое, включено по умолчанию только для чтения изображений при рендеринге
и обосновано тем же принципом, что и сетевой egress: куда пишется вывод —
решает вызывающий, и это видно прямо в вызове инструмента (`output_dir`);
какие локальные файлы читаются при рендеринге — решает содержимое
недоверенного документа, а вызывающий этот выбор никогда не делал.

## Архитектура

Сегодня `MarkdownToWordConverter` и `WordToMarkdownConverter` объявлены внутри
`mdtoword/app.py`, который в строках 7–14 импортирует PyQt6. Любой headless-потребитель
конвертеров тянет за собой Qt. Это блокирует MCP-сервер и уже мешает тестам:
`tests/test_markdown_converter.py` требует PyQt6 для проверки чистой логики рендеринга.

Ядро выносится в Qt-free модуль:

```
mdtoword/
  converters.py    НОВОЕ  ConversionError, MarkdownToWordConverter, WordToMarkdownConverter
  mcp_server.py    НОВОЕ  три инструмента поверх converters + workflow
  app.py           GUI; импортирует конвертеры, локализует ошибки
  gfm_renderer.py  без изменений
  latex_omml.py    без изменений
  workflow.py      без изменений
  theme.py         без изменений
```

Зависимости строго односторонние: `mcp_server.py → converters.py → gfm_renderer.py → latex_omml.py`.
PyQt6 не встречается ни в одном звене этой цепочки — это и делает сервер запускаемым
без дисплея.

### Контракт ошибок

Текущие конвертеры возвращают `(bool, str)`, где строка — это `"Ошибка при конвертации: ..."`,
а варнинги приклеены к тому же сообщению через `\n\n`. То есть локализация зашита в ядро,
а варнинги не отделимы от текста программно. Для батча в MCP это непригодно.

Новый контракт ядра:

- успех → возврат `list[str]` варнингов (возможно пустого);
- отказ → исключение `ConversionError` с сообщением без локали.

Локализация переезжает в `app.py`, где уже есть словарь `_text()` с RU/EN. GUI оборачивает
вызовы в `try/except ConversionError` и форматирует сообщение сам.

### Транспорт

Официальный Python SDK `mcp`, декораторный сервер, транспорт stdio.
Structured output — через Pydantic-модели.

**stdout — это сам протокол.** Любой `print` в цепочке импорта или выполнения рвёт сессию.
Логирование только в stderr. Защищено тестом.

### Установка

`pyproject.toml` в репозитории нет, console script без него не сделать, и заводить его
ради одного скрипта избыточно. Конфигурация клиента:

```json
{
  "command": "<repo>/.venv/bin/python",
  "args": ["-m", "mdtoword.mcp_server"],
  "cwd": "<repo>"
}
```

SDK `mcp` выносится в отдельный `requirements-mcp.txt`, а не в `requirements.txt`.
Причина: `requirements.txt` — источник истины для PyInstaller-сборки GUI и для
`test_packaging.py`, который сверяет его с `environment.yml`. Добавление туда серверного
SDK раздуло бы дистрибутив приложения и потребовало правки обоих файлов ради зависимости,
которая GUI не нужна.

## Инструменты

Общая форма отчёта у всех трёх — агенту не нужно учить три разные схемы.

| Инструмент | Параметры | Возврат |
|---|---|---|
| `markdown_to_word` | `inputs: list[str]`, `output_dir: str \| None`, `font_name: str = "Times New Roman"`, `font_size: float = 12`, `footnotes_heading: str = "Footnotes"`, `fetch_remote_images: bool = False`, `image_root: str \| None = None` | `ConversionReport` |
| `word_to_markdown` | `inputs: list[str]`, `output_dir: str \| None` | `ConversionReport` |
| `preview_markdown` | `inputs: list[str]`, `font_name: str = "Times New Roman"`, `font_size: float = 12`, `footnotes_heading: str = "Footnotes"`, `fetch_remote_images: bool = False`, `image_root: str \| None = None` | `PreviewReport` |

```
ConversionReport
  sources_found: int
  converted: list[{ source: str, output: str, warnings: list[str] }]
  failed:    list[{ source: str, error: str }]

PreviewReport
  sources_found: int
  previews: list[{ source: str, warnings: list[str] }]
  failed:   list[{ source: str, error: str }]
```

`preview_markdown` прогоняет полный рендер в память и не пишет ничего на диск. Нужен,
чтобы агент увидел предупреждения о непереводимых конструкциях (LaTeX, недоступные
изображения) и починил исходник до записи файла.

`inputs` принимает файлы и папки вперемешку: рекурсивный обход и фильтрация по
расширениям уже реализованы в `workflow.discover_sources`. Имена выходных файлов
выделяет `workflow.resolve_output_paths`, он же разводит коллизии внутри батча.

### Поведение на краях

- **`inputs` пустой** → `ValueError` с явным сообщением, а не пустой успешный отчёт; по протоколу MCP это в любом случае доходит до агента как `isError: true`.
- **`discover_sources` ничего не нашёл** → `sources_found: 0`, списки пустые. Поле
  существует именно затем, чтобы агент отличил «путь указан неверно» от «всё готово».
  Молчаливый пустой успех — худший из возможных ответов агенту.
- **Падение одного файла** не прерывает батч: файл уходит в `failed`, остальные
  конвертируются.
- **`output_dir` не существует** → создаётся с `parents=True`.
- **Существующий выходной файл** → перезаписывается молча (см. «Ограничение, принятое
  осознанно»).

### Описания инструментов

Описания — единственная документация, которую видит агент, поэтому в них прямым текстом:

- `markdown_to_word` поддерживает GFM, таблицы, сноски, изображения и LaTeX → OMML
  (нативные формулы Word);
- `word_to_markdown` **лоссовый**: извлекает заголовки по стилям `Heading N`,
  bold/italic по run'ам и таблицы; всё остальное — списки, изображения, формулы,
  сноски, стили — схлопывается в плоский текст.

Без явного предупреждения о потерях агенты будут гонять round-trip `md → docx → md`
и терять содержимое.

## Тестирование

- `tests/test_converters.py` — ядро без Qt. Существующий `tests/test_markdown_converter.py`
  переезжает сюда с импортом из `mdtoword.converters`; проверки не меняются.
- `tests/test_mcp_server.py` — in-memory клиент из SDK: схемы трёх инструментов,
  батч по папке, частичный отказ (один битый файл среди валидных), `sources_found: 0`
  на несуществующем пути, пустой `inputs`.
- Тест «импорт `mdtoword.mcp_server` ничего не пишет в stdout» — защита stdio-протокола.
- Регресс: остальные шесть наборов проходят без изменений.

## Что осознанно не делается

- Sandbox по разрешённым корням и флаг `overwrite` — отклонено при обсуждении
  как решение о доступе к файловой системе в целом. Узкий, включённый по
  умолчанию `image_roots` для чтения изображений при рендеринге — не
  пересмотр этого отказа, а отдельное решение из-за отдельной причины: см.
  «Локальные пути: какие изображения читает markdown_to_word» выше. Отказ от
  общего sandbox и подтверждения `overwrite` остаётся в силе.
- HTTP-транспорт — бессмысленен при контракте «только пути»: файловая система всё равно
  должна быть общей.
- Публикация отдельного пакета на PyPI — не нужна, пока потребитель один.
- Улучшение качества `word_to_markdown` — отдельная задача, не входит в этот спек.
  Здесь его лоссовость только документируется.
